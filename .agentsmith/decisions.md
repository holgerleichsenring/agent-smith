# Decision Log

## p0001: Core Infrastructure
- [Architecture] Command Pattern (MediatR-style) as central pattern ‚Äî strict separation of "What" (context) from "How" (handler)
- [Architecture] Single generic handler interface `ICommandHandler<TContext>` instead of multiple handler types ‚Äî enables DI-based dispatch
- [Tooling] YamlDotNet for configuration ‚Äî chosen for simplicity, avoids overengineering (no JSON schema validation at this stage)
- [Implementation] Lazy environment variable resolution in config loader ‚Äî unset vars aren't errors during loading, only at usage time
- [TradeOff] No validation of all environment variables upfront ‚Äî accepted incomplete config to gain flexibility

## p0002: Pipeline Handlers
- [Architecture] CommandExecutor uses `GetService<T>()` instead of `GetRequiredService<T>()` ‚Äî allows custom error messages instead of DI exceptions
- [Implementation] Handlers are sealed classes with primary constructors ‚Äî enforces immutability and modern .NET 8 conventions
- [TradeOff] LoadCodingPrinciplesHandler can be fully implemented in Phase 2 but intentionally stubbed ‚Äî accepted inconsistency to keep phases parallel
- [Implementation] ApprovalHandler auto-approved as stub ‚Äî establishes pattern for later headless mode

## p0003: Providers
- [Architecture] Separate provider for each source control system (GitHub, Local, GitLab later) ‚Äî each has different API, no shared abstraction
- [Tooling] ClaudeAgentProvider chosen as primary agent ‚Äî Anthropic API, agentic loop pattern established as reference
- [Architecture] ToolExecutor separated from agent loop ‚Äî allows code reuse across provider implementations
- [TradeOff] Scout phase (file discovery) designed but deferred to Phase 9 ‚Äî accepted complexity later to unblock core provider work
- [Security] Path validation in tools (no `..`, no absolute paths) ‚Äî prevents directory traversal attacks from agentic code execution

## p0004: Pipeline Execution
- [Architecture] Regex-based intent parser instead of LLM call ‚Äî chosen for determinism, cost, and Phase 4 speed
- [Architecture] CommandContextFactory maps command names to contexts ‚Äî avoids massive switch statement in PipelineExecutor
- [Implementation] ProjectConfig passed as parameter to PipelineExecutor, not injected ‚Äî enables per-run configuration variation
- [TradeOff] Intent parser intentionally simple ‚Äî can be swapped for Claude-based parser later without interface changes

## p0005: CLI & Docker
- [Tooling] System.CommandLine instead of manual argument parsing ‚Äî provides --help, validation, version support "for free"
- [TradeOff] No sub-commands, no daemon modes, no interactive shell ‚Äî "Simple: Input in ‚Üí PR out ‚Üí Exit"
- [Architecture] Multi-stage Docker build ‚Äî separates build (SDK) from runtime (slim), targets <200MB image size
- [Implementation] Config mounted as volume, not baked into image ‚Äî enables reuse without rebuild

## p0006: Retry & Resilience
- [Tooling] Polly via `Microsoft.Extensions.Http.Resilience` chosen over custom retry logic ‚Äî standard, battle-tested, reduces code
- [Architecture] Retry config as typed class, not raw numbers ‚Äî enables configuration without code changes
- [TradeOff] Conservative defaults (5 retries, 2sec initial, 2.0x backoff) ‚Äî balances safety against cost overruns from excessive retries
- [Implementation] Exponential backoff with jitter ‚Äî prevents thundering herd on rate limit recovery

## p0007: Prompt Caching
- [Architecture] `PromptCaching = AutomaticToolsAndSystem` for all API calls ‚Äî caches system prompt, tool definitions, coding principles
- [Implementation] Coding Principles positioned first in system messages ‚Äî maximizes cache prefix (longest stable content cached)
- [TradeOff] 5-minute ephemeral cache (API default) accepted ‚Äî balance between cost savings and context freshness
- [Architecture] TokenUsageTracker separates cache metrics tracking ‚Äî enables observability without coupling to loop logic

## p0008: Context Compaction
- [Architecture] Haiku for summarization, not custom rule-based compression ‚Äî LLM-generated summaries more useful than keyword extraction
- [Architecture] FileReadTracker deduplicates file reads ‚Äî prevents same file content from inflating conversation history
- [TradeOff] Last N iterations always preserved fully ‚Äî accepted overhead to maintain recent decision context
- [Implementation] Compaction triggered periodically, not continuously ‚Äî prevents overhead from constant summarization

## p0009: Model Registry & Scout
- [Architecture] TaskType enum maps task to appropriate model ‚Äî Haiku for discovery, Sonnet for coding, optional Opus for reasoning
- [Architecture] Scout as implementation detail of ClaudeAgentProvider, not separate interface ‚Äî keeps IAgentProvider stable
- [TradeOff] Scout is read-only (no write_file, no run_command) ‚Äî accepts limitation to prevent Scout from breaking code
- [Architecture] Backward compatible: Models config is optional ‚Äî existing single-model configs work unchanged

## p0010: Container Production-Ready

## p0096: Event Pollers
- [Architecture] Two independent Redis leases: `agentsmith:leader:poller` and `agentsmith:leader:housekeeping`. A single replica can hold both; having them separate means polling can go leader-elected without disturbing housekeeping timing, and vice versa. Same `LeaderElectedHostedService` runs both with different keys.
- [Architecture] LeaderElectedHostedService loop: TryAcquire ‚Üí while leader { work, renew, check } ‚Üí on renewal failure cancel work, release, back to try-acquire with 5s idle backoff. Ensures exactly-one-leader under Redis's TTL+CAS semantics.
- [Architecture] RedisLeaderLease uses CAS Lua for both RENEW (PEXPIRE only if stored value matches token) and RELEASE (DEL only on match). Prevents a GC-paused or network-partitioned leader from extending or deleting a lease the new holder acquired after the TTL elapsed ‚Äî the classic "Redlock lite" pattern.
- [Architecture] Housekeeping services (StaleJobDetector + EnqueuedReconciler from p0095c) migrated from unconditional-per-replica to leader-only. Reduces N-fold API calls; correctness was already there via atomic transitions, but the extra calls were wasteful. ServerCommand's RunHousekeepingAsync runs both services in one Task.WhenAll under the housekeeping leader.
- [Architecture] PollerHostedService fan-outs pollers via Task.WhenAll with a 20s per-poller timeout. One slow platform does not block the others. Exceptions caught per poller and logged; one failure does not poison the cycle. Claims from all pollers are then processed sequentially ‚Äî ClaimAsync is fast (few hundred ms worst case), and sequential processing keeps Redis lock contention predictable.
- [Architecture] IEventPoller reads via ITicketProvider.ListByLifecycleStatusAsync(Pending) ‚Äî same dependency the reconciler uses. p0096 ships the GitHub poller only; other platforms follow when their ListByLifecycleStatus implementations land (same incremental pattern as p0095c).
- [Architecture] Polling opt-in per project via PollingConfig (default Enabled=false). Webhook-only deployments see no behaviour change. Disabled projects are skipped at BuildPollers time in ServerCommand ‚Äî no null pollers flow into PollerHostedService.
- [Scope] Parallel-across-platforms, sequential-across-claims. Parallelism knob for ClaimAsync dispatch was considered but rejected ‚Äî claim is O(ms) unless a platform API is slow, and semaphore complexity here is not worth a few hundred ms.
- [Scope] Jitter is per-cycle (¬±10%) applied after Task.WhenAll. Per-project jitter (spec option) was simpler at this scope ‚Äî one randomised sleep at the end of each cycle, minimum across configured intervals.
- [Scope] Leader lease renewal interval 10s, TTL 30s. 3x safety margin matches the p0095c heartbeat pattern; if the process GC-pauses or clock-skews for up to 20s we still renew before the TTL hits.

## p0097: Parallel Skill Rounds
- [Architecture] DeferredBuffers context key as the channel between handler base and executor. SkillRoundHandlerBase always builds a `SkillRoundBuffer`; if the executor put a list under `ContextKeys.DeferredBuffers` before dispatching the batch, the handler appends to it and returns; otherwise it applies the buffer immediately (sequential default). Public `ICommandHandler.ExecuteAsync` signature unchanged ‚Äî no new interface, subclass handlers (SkillRoundHandler/SecuritySkillRoundHandler/ApiSkillRoundHandler) stay one-liners. Alternatives considered: (a) new `ISkillRoundCollector` returning `(CommandResult, SkillRoundBuffer)` ‚Äî adds plumbing to every handler; (b) explicit flag in PipelineContext queried by handlers ‚Äî same idea, more verbose. Picked the channel approach because presence/absence IS the signal.
- [Architecture] Observation IDs assigned at merge time, not parse time. `ObservationParser.ParseWithoutIds` returns `Id=0` for every entry; `ApplyBufferToContext` reads the runtime list, computes `nextId = Max(o.Id) + 1`, and stamps each buffer entry under the executor's serial merge order. Parallel completion order does NOT affect IDs ‚Äî graph order does. Cross-run ID stability is NOT claimed (LLMs vary observation count run-to-run); the contract is "deterministic within a single run."
- [Architecture] `PipelineBatchRunner` extracted from PipelineExecutor as a separate class (composition over fat orchestrator). Reason: PipelineExecutor was already over the class-size limit before parallelism arrived; adding fan-out + cancellation + buffer-merge would push it well past 500 lines. Extraction also models the runner as a real responsibility (parallel execution under throttle) instead of a private detail. PipelineExecutor now owns main loop + sequential single-step path + lifecycle; the runner owns SemaphoreSlim, linked CTS, and merge.
- [Architecture] `BatchOutcome` value object replaces a tuple-of-arrays return shape from the runner. The two queries operators care about post-batch ‚Äî first failure (fail-fast reporting) and first InsertNext (sequential follow-up insertion) ‚Äî sit on the outcome as named methods. Keeps PipelineExecutor's batch handler short and the intent legible.
- [Implementation] Buffer merge in graph order, not completion order. `MergeBuffersInGraphOrder` walks the original `LinkedListNode<PipelineCommand>` list and looks each skill name up in a `Dictionary<string, List<SkillRoundBuffer>>` built from the deferred list. Tasks may complete in any order, but merge order is fixed by the topology.
- [TradeOff] CancellationToken cancels orchestration but NOT in-flight LLM calls. Once the SemaphoreSlim releases and the inner `commandExecutor.ExecuteAsync` is past the point of return, Polly retries can still fire and Anthropic responses run to completion. Cost is recorded for every settled call (cancelled task's cost is tracked before the cancellation throws), so the cost report stays trustworthy. Operators should expect close-to-full batch cost even when one slot fails. Documented because "CTS = stop everything" is a reasonable misread.
- [TradeOff] `MaxCommandExecutions` (100) counts each command in the batch once. A 13-skill stage burns 13 of 100 in one parallel burst ‚Äî same total as today's sequential path, just consumed in fewer wall-clock seconds. Acceptable for now; revisit if security-scan pipelines with nested OBJECTIONs trip the guard.
- [Implementation] PipelineCostTracker switched from `Interlocked.Add`/`Increment` to a `lock`-based design. `Interlocked` covered the int counters but `_lastModel` write was already racy and `EstimateCostUsd`/`ToString` read multiple fields non-atomically. One lock simplifies the model and the contention window is negligible relative to the LLM call.
- [Implementation] PipelineContext `Dictionary<string, object>` wrapped under a single lock. The deferred-buffer pattern means most parallel writes go through the locked `List<SkillRoundBuffer>` in DeferredBuffers, but handlers still call `pipeline.Set(ContextKeys.ActiveSkill, name)` during execution. ConcurrentDictionary was considered but would force callers using `Get<T>` to handle missing-key differently. Single lock keeps the surface identical.
- [Architecture] Parallelism opt-in via `agent.parallelism.max_concurrent_skill_rounds` (default 1). Sequential path is byte-identical to pre-p0097: same logging, same execution trail, same observation IDs given identical LLM output. Verified by full test suite (1089 ‚Üí 1105, +16 new tests) all green at default.
- [TradeOff] Logging stays per-command in batch mode (each slot logs its own `[{Step}/{Total}] Executing ...` and `... completed`). With 13 parallel slots this can interleave on stdout ‚Äî tolerable in CLI mode, potentially noisy on Slack/Teams progress streams. Spec flagged this for verification; no batch-aware progress event added in p0097. Revisit if UX feedback on production runs (with `MaxConcurrentSkillRounds=4`) shows actual flooding.
- [Scope] InsertNext semantics on batch: first-in-graph-order InsertNext wins. If skills A and B in the same batch both return InsertNext, only A's follow-ups are queued. Conservative choice ‚Äî alternatives (concatenation, voting) considered too clever for the typical use case (OBJECTION from one skill triggering a single targeted re-round). Documented so future-self doesn't reinvent it.

## p0109: Jira Lock Decorator

- [Architecture] `IRedisClaimLock` is a multi-writer-coordination concern, not a Jira-platform concern. Cross-process serialization is needed only in the Server cluster (multiple pods, poller + webhook racing on the same ticket). A single CLI process cannot race with itself, and parallel CLI invocations have no shared Redis to coordinate through anyway. The lock attaches at composition time via a decorator (`LockedTicketStatusTransitioner`) that the Server registers and the CLI does not ‚Äî Jira's API atomicity gap (no If-Match on `fields.labels`) is real, but the *need* for the workaround is conditioned on the multi-writer reality, which is a deployment property, not a platform property.
- [Architecture] Decorator wraps Jira only, not all platforms. GitHub/GitLab use ETag/If-Match-equivalent semantics; AzDO uses `op:test` on `/rev` for optimistic concurrency at the API level. Wrapping them with the same lock would convert API-level retries into queueing ‚Äî strictly worse. The decorator class accepts any `ITicketStatusTransitioner` as inner type, but the `LockingTicketStatusTransitionerFactory` only applies it to `config.Type == "jira"`.
- [Implementation] DI rebind extracted from `AddRedis()` into a standalone `AddJiraLabelLockDecorator()` extension method. Reason: real Server `Program.cs` calls `AddRedis().AddCoreDispatcherServices()`, but `AddCoreDispatcherServices` internally calls `AddAgentSmithInfrastructure` which re-registers `ITicketStatusTransitionerFactory‚ÜíTicketStatusTransitionerFactory`. MS DI is last-wins for `GetService<T>`, so an in-`AddRedis` rebind would be silently overwritten by the subsequent infrastructure registration. Standalone method called *after* `AddCoreDispatcherServices` makes the override stick. Test fixture `ServerDiLifetimeTests` mirrors the same call order.
- [Implementation] `LockingTicketStatusTransitionerFactory` takes the **concrete** `TicketStatusTransitionerFactory` (not `ITicketStatusTransitionerFactory`). Reason: passing the interface would risk a self-referential cycle if the binding is rebound to Locking. Binding the concrete type explicitly via `sp.GetRequiredService<TicketStatusTransitionerFactory>()` makes the dependency unambiguous. Infrastructure registration adds `services.AddSingleton<TicketStatusTransitionerFactory>()` plus a forwarding lambda for the interface so resolves of either return the same instance.
- [TradeOff] `NullRedisClaimLock` deleted as a Pre-p0107 leftover ‚Äî it was never wired in production, conflicting with p0107's "no Redis-fallback in CLI" decision. Doc-comment-claimed use-case ("manual security-scan, fix without ticket lifecycle inject the dep but never call these methods") was based on the assumption that `IRedisClaimLock` had to be in the CLI graph at all; this phase removes that need entirely. `ServerDiLifetimeTests` fixture rebased onto `Mock.Of<IRedisClaimLock>()`. Sibling `Null*` classes in the same directory (`NullRedisJobQueue`, `NullJobHeartbeatService`, etc.) left untouched ‚Äî they're still used by the test fixture as DI test doubles, and removing them is out of scope.
- [TradeOff] `EnvVarCollection` xUnit collection introduced to serialize tests that mutate process-global env vars. Reason: new `LockingTicketStatusTransitionerFactoryTests` and `CliShapedDiTests` set `GITHUB_TOKEN`/`GITLAB_TOKEN`/etc. so the inner factory's `SecretsProvider` calls succeed; existing `FactoryTests.TicketProviderFactory_NewTypes_RecognizedButNeedSecrets` asserts the *opposite* (env unset ‚Üí ConfigurationException). xUnit's default per-class parallelism caused races. All three classes now share `[Collection(EnvVarCollection.Name)]`, and the env-mutating ones implement `IDisposable` to reset to clean state. Cheaper than introducing a `SecretsProvider` abstraction across the codebase for a test-only seam.

## p0109a: Server-Lifecycle Decoupling

- [Architecture] Coordinator (not decorator) for the lifecycle wrap. Lifecycle wrapping in `PipelineExecutor` is more involved than the Jira lock that p0109 handled with a decorator: it transitions a status, holds a heartbeat handle across the whole run, and disposes with a deferred close-transition. Hiding all of that behind `IPipelineLifecycleCoordinator.BeginAsync(...) ‚Üí IAsyncPipelineLifecycle` separates it cleanly from the executor's main loop, lets the CLI compose with `NoOpPipelineLifecycleCoordinator` (no Redis deps), and keeps the executor's ctor stable as new lifecycle concerns get added. Decorator pattern would have meant a wrapping `PipelineExecutor` ‚Äî too coarse a unit; the wrap belongs *inside* the executor's run, not around it.
- [Architecture] `ITicketClaimService` registration moved from Application to Server. Unlike `PipelineExecutor` (which CLI uses for non-ticket pipelines like apiscan/security-scan), `TicketClaimService` is purely Server-instantiated: only `WebhookRequestProcessor` and `PollerHostedService` resolve it, both Server-side. Application-side registration meant the CLI's DI graph carried `IRedisJobQueue` + `IRedisClaimLock` + `IJobHeartbeatService` as transitive deps even though no CLI code path can reach the service. Move resolves the transitive leak completely.
- [Implementation] `AddJiraLabelLockDecorator` renamed ‚Üí `AddServerCompositionOverrides`. The Server-side overrides on top of Application's CLI-safe defaults are now multiple: locking transitioner factory, ticket-aware lifecycle coordinator, ticket claim service. One method bundles them ‚Äî explicit "this is what Server adds beyond Application's CLI-safe defaults". The discipline is: Application/Infrastructure always register the Redis-free default; Server adds overrides on top.
- [Implementation] `NoOpPipelineLifecycleCoordinator` is the Application default. Reason: an Application/Domain default that's *safe everywhere* is the inversion of the Pre-p0107 pattern (Null fallbacks were registered conditionally based on Redis availability). Now: Application registers the no-op, Server unconditionally overrides. The Application graph is always resolvable in any composition. Test `Cli_PipelineLifecycleCoordinator_ResolvesToNoOp` asserts the default is in place.
- [TradeOff] `NullTicketClaimService` deleted ‚Äî same Pre-p0107 orphan pattern as `NullRedisClaimLock`. Only used by its own test, never wired in production. Doc-comment claimed "CLI server when Redis unavailable", which describes a fallback p0107 retired. With `ITicketClaimService` now Server-only, no CLI graph needs even an orphan dummy.
- [TradeOff] Real-CLI smoke test uses `AgentSmith.Cli.ServiceProviderFactory.Build` (the production composition) inside the test ‚Äî `Cli_RealServiceProviderFactoryBuild_ResolvesPipelineExecutor`. p0109's test only covered a hand-rolled CLI-shape collection, which missed the `IJobHeartbeatService` gap because it never resolved `PipelineExecutor`. The real-factory test ensures any future divergence between the test fixture and production composition is caught: if production registers something the fixture forgets, the test still fails because it builds from production code.

The default empty instance keeps the DI graph valid for tests and existing composition roots while leaving `PrimaryProvider = null`, which makes the override resolver a no-op (base SKILL.md always wins). Operators who want overrides register a populated AgentSmithConfig in their composition root before `AddAgentSmithCore` to win the last-binding race ‚Äî wire-up not done in this phase per the spec's "zero overrides ship" decision.
- [Implementation] **Frontmatter merge is shallow on top-level keys** per spec. `MergeFrontmatter` produces a new SkillMdFrontmatter where each top-level field takes the override's value if non-null, else base's. `activation`, `role_assignment`, `output_contract`, `references` are atomic ‚Äî declared in override means override wins entirely; omitted means inherit. The body is always taken from the override file (over.Body, not merged with base.Body). Reason: a deep merge inside `activation.positive[]` would silently double-list keys when both sides declare overlapping ones, and operators authoring an override are explicitly opting into a complete-declaration mental model.
- [Implementation] **`name` and `roles_supported` validation runs before the merge.** Override must declare both ‚Äî Name explicitly (non-empty), RolesSupported as a non-null list ‚Äî and both must match base exactly (Name byte-equality, RolesSupported set-equality). Mismatch throws `InvalidOperationException` with a clear message at parse time. Reason: a "sibling skill that happens to be a different skill" is not what an override is for ‚Äî those should live in different skill directories. Failing loud at boot prevents accidental identity drift.
- [Implementation] **INFO-level log when an override loads** ("Provider override loaded for skill '{Name}' from {Path}"); default behavior (no override matched) is silent. Operators see at boot which skills have provider-specific tuning without grep'ing files.
- [TradeOff] **8 tests instead of 7.** ProviderOverrideResolverTests adds a fourth case (`Resolve_NoActiveProvider_ReturnsBaseRegardlessOfOverrides`) beyond the spec's three ‚Äî guards the empty-string branch in ProviderOverrideResolver, which is the default behavior most operators experience. SkillMdParser tests deliver all 4 spec-named cases.
- [TradeOff] **Composition-root wireup deferred.** No CLI/Server entry point yet replaces the default empty AgentSmithConfig with the loaded one, so `primary_provider:` set in agentsmith.yml has no effect today. Reason: per spec, "Zero overrides ship in this phase. Provider-tuning is opt-in per skill, opt-in per operator." The mechanism is the deliverable; runtime wireup is a follow-up that lands when the first operator-authored override exists. ServerDi tests pass because the default empty config makes the resolver a no-op.

## p0112: Branch Persistence MVP (shipped as p0112a)

- [Architecture] **Push the work branch on every run that produced commits, regardless of pipeline outcome.** CommitAndPRHandler is a late pipeline step ‚Äî when a run fails before it (TestCommand fails, AgenticExecute crashes), local work in /tmp dies with the container. Push-on-failure makes the branch a durable artifact tied to the ticket, not the run. PR creation is decoupled from branch push: losing tokens because no PR was opened is the wrong trade-off.
- [Architecture] **Branch identity is `agent-smith/<platform>/<projectSlug>/<ticketId>`.** One branch per ticket, regardless of run count. Subsequent runs fast-forward to the existing remote branch instead of branching off main again ‚Äî avoids force-push by design. Hard cutover from legacy `fix/<ticketId>` per AAD-DEV operator confirmation; documented as a known break for any other deployment that needs a one-time migration script.
- [Architecture] **CheckoutSourceHandler resumes from remote branch when present** ‚Äî fetch + checkout if `agent-smith/<platform>/<projectSlug>/<ticketId>` exists; else create fresh from main. Triage / Plan / AgenticExecute see whatever state the previous run left. PlanArtifact-in-context (p0111c) lets a subsequent run short-circuit if available.
- [Implementation] **PersistWorkBranchHandler returns CommandResult.Fail with typed FailureKind on every error path** ‚Äî no log-and-return-Ok. PipelineExecutor wraps the persist invocation in its OWN try/catch separate from the normal command-loop catch: persist runs in the failure path AFTER the original failure cause is captured in `result`, so a persist exception (GitException, OutOfMemory in LibGit2 mid-call) must NOT overwrite the original failure cause for lifecycle.MarkFailed().
- [Implementation] **WIP commits prefixed `[wip] agent-smith run <runId>`** with `Run-Id`, `Pipeline`, `Failed-Step` trailers ‚Äî reviewers see at a glance the commit is mid-pipeline + non-human + log-correlatable. Auto-flips PR/MR to Draft on GitHub/GitLab; Azure DevOps logs INFO (no native Draft equivalent).
- [Implementation] **LibGit2Sharp reused, NOT shell-out git.** Existing GitHub/AzureRepos/GitLab/Local SourceProviders all use LibGit2Sharp; shell-out would diverge the codebase. Confirmed project-wide choice from p0001 onward.
- [TradeOff] **Push failures non-fatal warnings, not pipeline failures** ‚Äî pipeline already failed, can't make it worse. (project, ticket)-keyed counter increments on non-transient failures (auth misconfigured, branch protection blocked, divergent remote that won't resolve); after `ConsecutiveFailureThreshold=3`, escalates to error-level log + structured `persist-blocked` marker on the ticket lifecycle so recurring auth/permission issues don't get silently swallowed.
- [TradeOff] **Out-of-scope: container kills (OOM, eviction, SIGKILL, kernel panic).** Persist runs only when the pipeline cleanly handed control to PipelineExecutor's failure path ‚Äî a process crash dies before the handler. Documented as a known gap; mitigation belongs to a follow-up phase (periodic checkpoint-push during AgenticExecute, or sidecar/watchdog reading the running container's repo).

## p0114: OpenAI / Azure-OpenAI Context Compactor

- [Architecture] **Same conceptual algorithm as p0008's ClaudeContextCompactor, different SDK + caching semantics.** Anthropic exposes `cache_control: ephemeral` for verifiable prefix-caching; OpenAI/Azure use opaque automatic prompt caching above the 4096-token threshold without operator control. Compactor I/O shape is identical (message list in, summarized message list out); SDK glue + summarization-prompt tuning differ. New OpenAiContextCompactor handles `type=openai` + `type=azure-openai` (Azure delegates via the same composition pattern AzureOpenAiAgenticAnalyzer uses); `type=gemini` + `type=ollama` get NoOpContextCompactor as an EXPLICIT placeholder, NOT a denial that they need it ‚Äî those deserve their own implementations in follow-up phases.
- [Architecture] **Trigger is boolean OR**, written as a Boolean expression to remove ambiguity: `compactor.ShouldFire = (currentIterations >= ThresholdIterations) || (estimatedAccumulatedInputTokens >= MaxContextTokens)`. Either alone fires. Defaults from existing CompactionConfig: `ThresholdIterations=8, MaxContextTokens=80000`. Operators set them once per project; both compactors honor the same knobs.
- [Architecture] **Compaction point fires AFTER an assistant turn produces no further tool_calls** ‚Äî between rounds, never mid-round. Inline comment in OpenAiAgenticLoop + OpenAiAgenticAnalyzer at the call site documents this so future parallelization of the loop has to explicitly handle the constraint. Cheap insurance against subtle breakage.
- [Architecture] **Scoped to agentic loops only**, not the simple CompleteAsync paths (TriageHandler, GeneratePlan first call, ContextGenerator). Single-shot calls don't accumulate history; the agentic loop is where the message list grows turn-on-turn.
- [Implementation] **Pre-compaction token count is an ESTIMATE (tiktoken-equivalent, ¬±5%), used only for the trigger threshold.** Post-compaction reports VERIFIED `usage.prompt_tokens` from the next response ‚Äî the API's authoritative count. CompactionEvent records both: `tokens_saved = previous_response.prompt_tokens - next_response.prompt_tokens` once both are known. Trigger uses estimate; reporting uses ground truth.
- [Implementation] **Tail-boundary respects tool-call pairs.** OpenAI rejects requests where a `tool` message references a `tool_call_id` without the matching prior `assistant(tool_calls=[...])` in the same array. Definition: tail = last N COMPLETE rounds (assistant.tool_calls + matching tool messages); compactor walks backward identifying boundaries, never splits mid-pair. Default `N=2`. Bare assistant text reply (no tool_calls) counts as one degenerate round with no tool tail.
- [Implementation] **Summarization prompt via IPromptCatalog (p0103c)** ‚Äî consistent with all other prompt resources. New resource `src/AgentSmith.Application/Prompts/Resources/openai-context-compactor-system.md`. CompactionEvent logs SHA-256 prefix (8 hex chars) of the resolved prompt for drift detection between runs without dumping the prompt itself.
- [Implementation] **`CompactionConfig.DeploymentName` (string?, default null) routes summarization to a cheaper deployment** (e.g. `gpt-4o-mini-deployment`). Compaction is summarization ‚Äî running it on the project's gpt-4.1 Primary deployment burns the same per-token rate as the actual work. Default behavior unchanged when unset; no breaking config migration.
- [TradeOff] **Compactor failure (rate-limit, error, garbage summary) is non-fatal** ‚Äî falls back to original messages with WARN log + `CompactionEvent.Failed=true`. Agentic loop continues with un-compacted history. Pretending compaction succeeded with a broken summary is worse than running on the full history.
- [TradeOff] **Compaction itself costs tokens** ‚Äî one extra LLM call per event, charged against the configured deployment. Net win when conversation > ~50k tokens; below that the trigger doesn't fire (`CompactionConfig.IsEnabled` gate plus the threshold). Operators can disable per project for traceability in regulated environments.

## p0115: AgentSmith.Sandbox.Agent + Carrier Image (Init-Container Pattern)

- [Architecture] **Init-container pattern with ONE carrier image, NOT pre-baked toolchain images.** The original spec draft listed `sandbox-dotnet`, `sandbox-angular` etc. as pre-baked images we'd publish. Corrected before implementation: we ship exactly one image (`agent-smith-sandbox-agent`) whose only job is to deliver the agent binary into the sandbox pod. The Server pod (p0116) constructs each sandbox pod with two containers sharing an `emptyDir` volume: an initContainer using our carrier image (its ENTRYPOINT copies `/agent` to `/shared/agent` and exits 0), and a main container using the unmodified upstream toolchain image of choice (`mcr.microsoft.com/dotnet/sdk:8.0`, `node:20`, `mcr.microsoft.com/openjdk/jdk:21-ubuntu`, ‚Ä¶) with `command: ['/shared/agent']`. We maintain ONE image; toolchain updates flow automatically from upstream; operator-custom toolchains work without our involvement; new languages need zero changes from us.
- [Architecture] **Self-Inject mode (Option B), not a Busybox stage (Option A).** The carrier's ENTRYPOINT is the agent itself with `--inject <target>`. The agent's `BinaryInjector` resolves `Environment.ProcessPath` and copies the binary to the requested target with executable permissions, then exits 0. Rejected alternative ‚Äî a separate Busybox stage providing `/bin/cp`: would add 1 MB and a second tool to the carrier for a 30-line copy operation that the agent can do natively. Self-inject also keeps the carrier truly single-binary.
- [Implementation] **Carrier base = `mcr.microsoft.com/dotnet/runtime-deps:8.0-bookworm-slim`, NOT distroless.** First attempt used `gcr.io/distroless/cc-debian12` (tight, small) but it lacks `libz.so.1` which self-contained .NET 8 needs at startup. `gcr.io/distroless/base-debian12` lacks `libgcc` and `libstdc++` too. Microsoft's `runtime-deps` is the documented minimal base for self-contained .NET and bundles everything required (glibc, libgcc, libstdc++, libz, libssl, ICU stubs). Cost: image is ~218 MB instead of the spec's ~80 MB target. That's the price of running a real .NET binary in a real container; distroless ergonomics don't apply here. Documented in the Dockerfile comment block.
- [Implementation] **`IncludeAllContentForSelfExtract=true`, NOT `IncludeNativeLibrariesForSelfExtract=true`.** Spec said "same pattern as p52 used for the CLI" with `IncludeNativeLibrariesForSelfExtract=true`. That setting bundles native libs for self-extraction at startup but keeps managed assemblies in-bundle. In practice this caused CoreCLR startup to fail with "Failed to load System.Private.CoreLib.dll" because the TPA list ended up empty (`COREHOST_TRACE=1` confirmed: `TRUSTED_PLATFORM_ASSEMBLIES = `, `CoreCLR path = ''`). Switching to `IncludeAllContentForSelfExtract=true` extracts the entire bundle to `$DOTNET_BUNDLE_EXTRACT_BASE_DIR` (defaults to `/tmp/.net/...`) on first launch; CoreCLR then loads cleanly. Cost: a one-time disk write (~75 MB) per fresh container startup. Acceptable for the pod lifetime model ‚Äî extraction stays around for the full pipeline.
- [Implementation] **ProcessRunner catches `Win32Exception`/`InvalidOperationException` from `Process.Start` and translates them to `ProcessOutcome(-1, false, message)`.** Original spec didn't address this ‚Äî Process.Start throws when the target binary or working directory doesn't exist, which would propagate through StepExecutor ‚Üí JobLoop ‚Üí Program and crash the agent (exit 3) on the first bad step. Smoke test caught this immediately (`echo` against missing `/work` directory crashed the loop). The catch turns the failure into a normal step failure (`StepResult` with ExitCode=-1 + descriptive ErrorMessage), letting the loop continue. Aligns with the spec's "Run steps without a Command are pushed as a step failure (ExitCode=-1) and the loop continues" ‚Äî same shape, broader coverage. Two new tests (`RunAsync_NonexistentCommand_*`, `RunAsync_NonexistentWorkingDirectory_*`) regression-guard the behavior.
- [Implementation] **Three classes for what the spec called "StepExecutor", not one.** Spec described a single `StepExecutor` doing process spawning + soft-batching + timeout enforcement. That would be ~150 lines violating the 120-line class limit. Split into `ProcessRunner` (Process.Start + readers + timeout + kill), `OutputBatcher` (thread-safe buffer + 50/100ms flush), and `StepExecutor` (orchestrator: validates, wires the two together, emits Started/Completed events, packages result). Each is single-responsibility and ‚â§90 lines. Same external contract: `IStepExecutor.ExecuteAsync(step, onEvents, ct)`.
- [Implementation] **Two interfaces (`IRedisJobBus`, `IStepExecutor`) extracted purely for testability.** Spec didn't list them. Without them, `JobLoop` would be coupled to two concrete classes, blocking unit tests via `Mock<>`. The interfaces have one consumer each, so it's a marginal abstraction cost for full unit-test reach.
- [Implementation] **Wire-format JSON uses string-cased enums (camelCase: `"run"`, `"stdout"`, ‚Ä¶) via `JsonStringEnumConverter(JsonNamingPolicy.CamelCase)`.** Numeric enums would round-trip fine but render as integers in `redis-cli LRANGE` / `XRANGE` ‚Äî which is exactly the debugging tool the spec called out as a reason for picking JSON over Protobuf. Strings make Redis snapshots human-readable. Cost: ~3 bytes per event. Centralized in `WireFormat.Json` so both sides of the wire (agent + future Server pod in p0116) deserialize the same way.
- [Implementation] **Step backward-compat for missing `kind` field uses default-value parameters on the positional record.** Spec said "System.Text.Json deserializes a missing enum to the 0 value = Run" ‚Äî that only works if the record parameter has a default. Implemented as `StepKind Kind = StepKind.Run, ‚Ä¶ int TimeoutSeconds = Step.DefaultTimeoutSeconds`. A missing `kind` field on incoming JSON now deserializes to Run; missing `timeoutSeconds` defaults to 600. Required fields (`schemaVersion`, `stepId`, plus `command` for Run validated by JobLoop) still throw if absent.
- [Implementation] **WaitForStepAsync polls `LPOP` with 250 ms sleep, NOT `BRPOP`.** Spec mentioned BRPOP/LMPOP. StackExchange.Redis is a multiplexer and doesn't expose true blocking calls (BRPOP would tie up the connection). Polling LPOP with a short sleep is the canonical workaround under StackExchange.Redis and avoids the multiplexer-blocking trap. The 250 ms tick is invisible compared to step durations (seconds-to-minutes).
- [Tooling] **Separate test project (`AgentSmith.Sandbox.Agent.Tests`), not added to the existing single `AgentSmith.Tests`.** The repo convention so far has been one consolidated test project. The sandbox agent is meant to be a standalone deployable, and Testcontainers.Redis is a heavy new dep that would slow every unrelated test discovery. The new project keeps the agent's test surface isolated and the existing 1338-test suite untouched. Confirmed with the user before splitting.
- [Tooling] **NO `ProjectReference` to `AgentSmith.Contracts`.** Spec listed Contracts as a possible reference "for constants / logging setup if needed". In practice the agent needs nothing from Contracts ‚Äî its 4 model types (Step/StepEvent/StepResult/RedisKeys) are all new and self-contained. Skipping the reference (a) keeps the self-contained binary lean by avoiding Contracts' transitive dependency on Domain plus all its types, and (b) proves the agent really is decoupled. Confirmed with the user.
- [Tooling] **CI publish gated to `v*` tags only, NOT main pushes.** Per user direction. The matrix entry for `agent-smith-sandbox-agent` builds on every event (validates the Dockerfile in PRs and on main) but only pushes to Docker Hub when a v* tag is created. Lifts to "always" once the Docker Hub repo + description are ready. Implemented via a `publish: tags` matrix key + per-step `if:` on the build-and-push step.
- [Security] **Git credentials flow via pod env vars, NOT via `step.env`.** This decision pre-existed in the spec but is worth highlighting: anything in `step.env` lands in the Redis input stream and is readable via `redis-cli`. Pod env is only visible via `kubectl describe` (RBAC-gated). The agent passes pod env through to subprocess automatically (Process.Start inherits parent env), so a `git clone` step using `credential.helper='!f() { echo "username=x-access-token"; echo "password=$GIT_TOKEN"; }; f'` reads the token from pod env without it ever touching Redis. The carrier image deliberately leaves USER unset so the Server pod controls runtime UID/GID per pod (typically matched to the toolchain main container via `fsGroup`).

- [Architecture] **Wire records extracted to a new `AgentSmith.Sandbox.Wire` project. - [Architecture] **`InProcessSandbox` lives in `AgentSmith.Infrastructure`, NOT `AgentSmith.Server` as the original spec listed.** CLI does not (and must not) reference Server. Putting the in-process implementation in Infrastructure lets both CLI and Server pull it. All 4 providers (Claude / OpenAI / Azure-OpenAI / Gemini / Ollama) implement the 8-arg signature; `AzureOpenAiAgentProvider` inherits from OpenAI so doesn't need its own override.
- [Architecture] **`ToolExecutor` keeps a dual-mode constructor: optional `ISandbox?` argument routes file/list/run tools through `SandboxToolHandler` when set, falls back to direct repository-path access (legacy `FileToolHandler` + `CommandRunner`) when null.** Reason: `ScoutAgent` and other non-pipeline callers continue to operate on the server filesystem during the migration. - [Architecture] **`run_command` tool wraps the LLM-supplied command in `Step{Command='/bin/sh', Args=['-c', commandLine]}`.** The LLM tool-schema continues to accept a single command STRING with shell pipes/redirects/quoting; without `sh -c`, `ProcessStartInfo.ArgumentList` would silently break any pipe like `cat foo | grep bar`. The wrap preserves existing skill prompts unchanged.

Per the explicit Option C scoping: minimum-viable demo path is K8s + InProcess + Wire + the three core handlers. Everything else is post-demo polish.

ProjectMap is the canonical source after p0110b; no-ProjectMap ‚Üí skip with a clear log line.
- [Implementation] **`SandboxBackendInfo` record wraps the `SandboxBackend` enum for DI registration.** `services.AddSingleton(backend)` failed because `AddSingleton<TService>(TService)` requires a reference type ‚Äî enums are value types. Wrapping in a record keeps the registration straightforward and exposes the chosen backend to startup logging.

Operators with unusual UIDs override via `SandboxSpec.SecurityContext`.

## p0124: Tight SkillObservation contract + batched filter with resilient parsing

- [Architecture] **`SkillObservation.Description` capped at 500 chars; new optional `Details` field at ‚â§4000 chars carries long-form prose.** Empirical sweep of recent runs showed typical Description is 200-300 chars, worst case ~500 ‚Äî the 1500-char cap initially proposed was inflated theory. 500 enforces brevity for Console/Summary scanability; multi-paragraph rationale moves to Details, which Markdown/SARIF render but Console/Summary do not. `ObservationCaps` constants are the single source of truth (DescriptionMaxChars=500, SuggestionMaxChars=300, RationaleMaxChars=500, DetailsMaxChars=4000).
- [Architecture] **Channel-aware output rendering.** Console ‚Üí Description-only; Summary ‚Üí Description-only; SARIF.message.text ‚Üí Description (full), SARIF.properties.detailed_message ‚Üí Details (when present); Markdown ‚Üí Description as section head + Details as body paragraph when present. Reason: Console/Summary serve fast triage, Markdown/IDE-hover serve archival reading ‚Äî same observation, different channels. Per the user's framing ("die Texte k√∂nnen gerne komplett in die markdown.md aber nicht in die console und auch nicht in die summary").
- [Architecture] **Field truncation with marker, not rejection.** Over-cap fields get truncated to `cap - markerLen` then suffixed with `‚Ä¶[truncated, original was N chars]`. Reason: rejecting drops real findings on a length technicality; truncating preserves the headline + emits a per-(skill,field) warning so skill authors see the nudge. - [Architecture] **Filter prompt input excludes Details.** The yes/no filter decision needs the headline, not multi-paragraph context. Including Details would inflate per-obs filter input from ~1500 to ~5500 chars, collapsing batch_size from 18 to 5 at max_output_tokens=8192. New private static helper `FilterRoundHandler.RenderForFilter(observation)` emits the trimmed shape (omits Details and Rationale); existing observation-render path stays for non-filter contexts. Filtered-output observations preserve Details intact ‚Äî we just don't ship it INTO the filter prompt.
- [Architecture] **Batched filter with deterministic batch size from caps, not heuristic budget fraction.** `batch_size = floor(max_output_tokens / 375) √ó 0.85` (worst-case 1500 chars per obs ‚âà 375 tokens; 0.85 reserve for LLM verbosity + JSON envelope). At 8192 ‚Üí batch_size ‚âà 18; at 4096 ‚Üí ‚âà 9. Worst-case basis (not realistic average): if a skill ever emits all-fields-at-cap, we still fit. With caps known, the formula has a single source of truth ‚Äî the previous 2/3-budget-fraction was guessing because the cap was unbounded. `TokenBudgetBatcher.Split` does the splitting via JSON-serialize-and-measure (no tiktoken dependency, 4 chars/token proxy).
- [Architecture] **Per-batch failure mode is best-effort + visible.** When a batch returns zero parseable observations, that batch's input observations are kept unchanged and `failedBatchCount` increments. After all batches, if `failedBatchCount > 0`, `FilterCoverageObservationFactory` appends a single Coverage-Incomplete observation: Concern=Correctness (meta-info, not a security finding), Severity=Info, Category=`"meta"`, Description = `Filter coverage incomplete: X of Y batches unparseable; Z observations not reviewed and may include false positives.`, ReviewStatus=`"not_reviewed"`. Severity=Info is already excluded from HIGH/MEDIUM/LOW counts in `ObservationSummary`, so it shows up in Markdown/SARIF without polluting the headline metric.
- [Architecture] **`ResilientJsonObjectExtractor` is the safety net for genuine output-token truncation.** Caps make truncation-mid-array unlikely (batches deterministically fit), but LLMs can still over-emit (verbose preamble pushing the actual JSON past the budget). Brace-counting state machine with string-literal awareness (`"` toggles in-string, `\` escapes next-char) ‚Äî ~30 lines, no state-machine library, no Utf8JsonReader complexity. Right separation: extractor finds object boundaries, JsonDocument.Parse validates content per-block.
- [Architecture] **`ObservationParser.TryParseWithoutIds` gains a resilient fallback path.** Order: try `ExtractJsonArray` + `JsonDocument.Parse` (current strict path). On `JsonException`, fall through to `ResilientJsonObjectExtractor.ExtractObjects` on the raw response ‚Üí element-wise `TryBuildObservation` per extracted block ‚Üí return survivors. If that yields zero, return null (true full-failure). The strict path is fast for well-formed responses; the resilient path is the diagnostic / recovery channel.
- [Implementation] **Per-batch logging surfaces input count + expected output chars + actual response chars.** The 0.85 reserve is observable in logs, not just guessed. If actual response chars consistently approaches expected, operators see the headroom is being eaten and can lower batch_size or raise max_output_tokens. If actual is consistently well below expected, the reserve was over-conservative and could be tightened in a future tuning pass.
- [Implementation] **Single-observation-exceeds-budget edge case is no longer reachable.** With Description ‚â§500 and other fields capped, the worst-case JSON-serialized observation is bounded at ~1500 chars. At any reasonable max_output_tokens (‚â•2000), an observation always fits a batch. Even if a skill bug emitted a 4000-char Description, the parser truncates to 500 with marker, so the persisted observation is still ‚â§500. Batching arithmetic always works ‚Äî no special-case for oversized observations needed.
- [Tooling] **agent-smith-skills 1.6.0 ‚Üí 1.7.0 is the recommended pairing, but pre-1.7.0 skills still work via parser truncation.** Hard-blocking on a specific skills version would brick existing operator setups when they upgrade agent-smith. The parser cap-truncation path enforces correctness regardless of skill version. With v1.6.0 skills the warning logs are noisier (every long Description gets truncated); with v1.7.0 they're rare. Operators running agent-smith p0124 against v1.6.0 skills will see a recurring "Description truncated from N chars" warning until they upgrade ‚Äî that's the migration nudge. All 16 api-security skills get the standardized length-contract paragraph in their `## Output` section.
- [TradeOff] **No tiktoken-based size estimation.** JSON-serialize-and-measure plus the 0.85 reserve handles the slack without a token-counting dependency. The 4-chars-per-token proxy is conservative for English JSON output (real ratio is closer to 3.5-4.0). Adding tiktoken would tighten the estimate but adds a per-provider tokenizer dep we'd have to keep in lockstep with the model ‚Äî not worth it for a 15% reserve we already build in.
- [TradeOff] **No retry on per-batch failure.** The batch+fallback combo delivers usable output without retry; retry adds cost (additional LLM calls) and complexity (idempotency, retry budget) for marginal benefit. Coverage-Incomplete observation makes the gap visible ‚Äî operator can re-run if they care.
- [TradeOff] **Per-Concern field-cap overrides deferred.** Current 16 api-security skills don't need it; legal-analysis pipeline can revisit when it hits the cap. The `ObservationCaps` shape is forward-compat ‚Äî adding per-Concern overrides later is additive, not breaking.
- [TradeOff] **No artifact-mode batching.** Artifact-role filters produce a single text artifact, not a list; size pressure hasn't surfaced there. If it does, the batching shape can be lifted into the artifact-mode handler later.

## p0124 follow-up: TriageOutputValidator vocab-fallback

- [Architecture] **`TriageOutputValidator.HasKey` accepts any key declared in `concept-vocabulary.yaml` as a valid rationale, even when the cited skill author didn't list it in `role_assignment.<role>.positive`.** Skill-specific keys remain useful as priority signals to the triage prompt (highest-fit skill picks given run context), but they're no longer a closed enum the LLM must memorize. Vocab is the closed namespace; if a key is in vocab, it's a known concept and the LLM's choice is semantically valid. **Why:** during a real api-security-scan run after agent-smith-skills 1.7.1 (which added 15 vocab keys + activation/role_assignment metadata to all skills), the LLM picked `api_security_scan` as the Lead-rationale for `api-vuln-analyst` ‚Äî vocab-valid, semantically applicable ‚Äî but the validator rejected because `api-vuln-analyst.role_assignment.lead.positive` only lists `nuclei_primary` and `owasp_categorisation`. Even after retry, the LLM converged on the same "wrong" choice and triage failed hard. Vocabulary-skill drift was breaking triage with no recovery path. **How to apply:** the skill's own `role_assignment.<role>.positive`/`negative` keys are still checked first (priority signal), the existing `HasNoCriteriaAtAll` defensive default still applies, then vocab-fallback. Order: skill-activation ‚Üí skill-role-assignment ‚Üí no-criteria-defensive ‚Üí vocab. Keys outside both skill criteria AND vocab remain rejected ‚Äî that's a true LLM hallucination.
- [Implementation] **Vocabulary plumbed into pipeline context as `ContextKeys.ConceptVocabulary`.** `ISkillLoader` gains a `LoadVocabulary(string skillsDirectory)` method (returns `ConceptVocabulary.Empty` when no vocab file found). `LoadSkillsHandler` calls it alongside `LoadRoleDefinitions` and stores both. `TriageOutputProducer` reads vocab from pipeline context and passes it to `validator.Validate(...)`. The validator's `Validate` overload accepts `ConceptVocabulary?` (nullable, defaults to Empty for backward compat with existing callers and test mocks). The original `Validate(output, skills)` overload is removed since callers were a closed set.
- [TradeOff] **Strict `role_assignment.<role>.positive` validation deliberately removed in favor of soft-priority-signal semantics.** Tighter validation theoretically catches more LLM hallucinations early, but in practice the LLM picks vocab-valid keys that just happen not to be in the skill's narrow whitelist ‚Äî and the validator can't distinguish "smart fallback" from "hallucination" without the vocab as the reference set. The robust answer: trust vocab as the closed namespace, treat skill criteria as priority hints. Skill authors writing role_assignment blocks now describe *which keys this skill responds to most strongly* (used by the prompt for fit-scoring), not *which keys the LLM is allowed to cite when assigning this skill* (used by the validator as a closed enum).

## p0125b: Boolean Expression Evaluator + Run-State (D1 slice 2/4)

- [Architecture] **AST records are pure data; evaluation lives in a separate ActivationEvaluator service.** Putting `Evaluate(state)` on each AST record means a virtual base method and behavior on data ‚Äî small inheritance, but still inheritance. A switch-expression in `ActivationEvaluator` keeps records as immutable values, centralizes evaluation logic for one-glance review, and makes "add a new node type" a build-error-driven update. Trade-off: one indirection at the call site, worth it to satisfy the composition-over-inheritance principle. -- alternatives: virtual `Evaluate(state)` on each AST record (rejected: behavior on records, breaks records-for-DTOs).
- [Architecture] **Parser split into three ‚â§120-line classes: ActivationExpressionTokenizer + ActivationExpressionParser + ActivationEvaluator.** A single class covering tokenize+parse+evaluate would be ~250 lines. Splitting along the natural seams (lex ‚Üí parse-to-AST ‚Üí evaluate-AST) gives three single-responsibility classes that test independently. Within Parser, per-precedence helpers (ParseOr ‚Üí ParseAnd ‚Üí ParseNot ‚Üí ParseComparison ‚Üí ParseAtom) are private methods of 5-15 lines each.
- [Architecture] **Comparisons (`=`, `>`, `>=`, `<`, `<=`) bind tighter than NOT.** Matches conventional boolean-algebra reading: `findings_count > 0 AND NOT api_internal_only` parses as `((findings_count > 0) AND (NOT api_internal_only))`. This is also Python/SQL convention. -- alternatives: NOT tighter than comparison (rejected: would force redundant parens around every `findings_count > 0` in skill activates_when fields).
- [Architecture] **Evaluator infers the comparison type from a literal operand, not the vocabulary.** `findings_count > 0` ‚Üí IntLit on the right ‚Üí call `state.GetInt("findings_count")`. `pipeline_name = "fix-bug"` ‚Üí StringLit on the right ‚Üí call `state.GetEnum`. The runstate impl validates the type against the vocabulary internally, throwing `ConceptTypeMismatchException` if the identifier's declared type clashes. The evaluator stays vocabulary-agnostic, which keeps the IRunStateConcepts contract narrow (no `GetType(name)` method needed). -- alternatives: pass `ConceptVocabulary` into the evaluator so it can dispatch on declared type (rejected: pushes vocabulary lookup into a service that already has a state surface; doubles the responsibility).
- [Implementation] **String comparisons are equality-only; ordered string comparison throws at evaluation time.** Ordered string comparison is locale-sensitive and surprising in activation expressions (`'apple'` > `'banana'` depends on culture). Equality is unambiguous. If lexicographic comparison is ever needed, the type system can grow a dedicated comparison operator; until then this stays banned. Same rule for bools.
- [Implementation] **`ActivationExpressionParseException` carries `Offset` + `OffendingToken`, not just a message.** Parse errors on hand-written skill frontmatter are operator-debuggable; a "parse failed" string with no location wastes their time. Position is byte-offset into the original input (column-style); offending token is the raw source slice. The CLI verb in p0125d will format these into per-skill error messages.
- [Implementation] **`IRunStateConcepts.SetX` throws on type-mismatch at the call site, not on the next read.** Writers fail loud. Silent coercion would let a handler accidentally publish a bool-shaped value to an int concept and have triage downstream make a wrong activation decision with no signal in the call chain. `ConceptTypeMismatchException` carries (concept name, declared type, attempted type) so the error message points at the bug.
- [Implementation] **ConceptValues storage is one `Dict<string, object>` under a single `ContextKeys.ConceptValues` key, not per-type dictionaries.** Per-type dicts (Dict<string,bool>, Dict<string,int>, Dict<string,string>) split the read/write surface across three lookups and force `PipelineContextRunStateConcepts` to know which dict each concept lives in via the vocabulary. Single dict with `object` values is one lookup; the typed cast happens in the Getter against the vocabulary-declared type. Boxing cost is negligible (concept reads happen at triage time, not in hot loops).
- [Implementation] **`PipelineContextRunStateConcepts` registered via a `Func<PipelineContext, IRunStateConcepts>` factory delegate, not as a direct DI binding.** PipelineContext is created `new` per pipeline run by `ExecutePipelineUseCase`; it's not in the DI container. Direct registration would fail at resolution time. The factory delegate is registered as a singleton in Infrastructure DI; it pulls the vocabulary from the context's ConceptVocabulary slot (falling back to `ConceptVocabulary.Empty` for test fixtures and dev environments without a loaded skills set) and constructs the impl. Future handlers (p0125c) inject `Func<PipelineContext, IRunStateConcepts>` and call it in their command handler. -- alternatives: register the type with PipelineContext as a constructor dep (rejected: PipelineContext isn't a DI service).
- [TradeOff] **Parser has no caching, no instance state.** `ActivationExpressionParser.Parse(string)` is a pure function; the class only exists for DI registration consistency. Parsing happens at boot (CLI validate-concepts) or at skill-load (when p0127 lands). No request-path parsing, no cache invalidation problem. If parsing ever shows up in a hot path, a memoizing decorator wraps the parser without changing the contract.
- [TradeOff] **Identifier-to-identifier comparison is rejected at evaluation time.** No skill activates_when uses it today (every example pairs an identifier with a literal). Implementing it would require either adding `GetType(name)` to `IRunStateConcepts` or threading vocabulary into the evaluator; both expand the surface for a hypothetical caller. If the need surfaces, the evaluator can grow this case alongside a `GetType` lookup, additively. -- alternatives: support it now via vocabulary lookup in the evaluator (rejected: premature, doubles evaluator responsibility).

## p0128c: opt-in data-flow gating (D4 slice 3/3)

- [Architecture] **`PipelineContext` got an opt-in `IPipelineContextReadGate` hook instead of being subclassed.** PipelineContext is `sealed` and consumed by every handler. A wrapper subclass would force a per-handler choice (which class to take); a separate `GatedPipelineContext` with its own surface would mean every handler checks both types. The hook stays minimal: PipelineExecutor `using (context.AttachReadGate(gate)) { ... }` scopes the gate around `SafeExecuteAsync`. Internal executor reads (e.g. ResolvedPipeline lookup before the gate attaches) bypass the gate naturally. -- alternatives: unseal PipelineContext (rejected: breaks cleanliness ‚Äî the type's intent is to be a flat state bag, not a polymorphism axis); thread a separate IPipelineContextReader through every handler (rejected: 100+ touchpoint refactor for a feature that defaults to warning-only).
- [Architecture] **`DataFlowReadGate` only fires on `Get`/`TryGet`. `Set` is unmodified.** The producer writes freely; the consumer's read is the gate. This matches the spec's "wrapper rejects reads of keys not declared by the active phase's incoming edges" and keeps the write path entirely uninstrumented. -- alternatives: gate writes too (rejected: producers can write whatever they like ‚Äî the contract is what consumers can READ); gate only Get and not TryGet (rejected: TryGet is the more common pattern in handlers, would create huge gap).
- [Architecture] **`PermissivePhaseDataFlow` baseline ships ONE wildcard edge `("*", "*", ["*"])`.** Spec said "every existing PipelineContext.Get<T>/TryGet<T> call site that reads a key produced by an upstream phase either gets covered by the declared edges or refactors". With 165 reads across the Application layer, a strict per-key declaration is days of work and high-regression-risk; the wildcard baseline keeps `enforce=true` viable from day one without breaking anything, and follow-up phases tighten edges per preset as the data-flow story crystalizes. The gate infrastructure is fully load-bearing (DataFlowViolationException, warn vs throw, JSON Pointer errors); only the per-preset edge declarations are placeholder. -- alternatives: declare every edge precisely now (rejected: scope creep ‚Äî would have doubled this phase's size; spec already allows enforce=false default to defer this); ship without preset declarations and rely on resolver-fallback bypass (rejected: violates done criterion "every preset has a matching IPhaseDataFlow").
- [Implementation] **Wildcards: `FromPhaseStep == "*"` matches any producer; `ToPhaseStep == "*"` matches any consumer; `ContextKeys` containing `"*"` matches any key.** The three wildcard axes compose, so an edge `("*", "*", ["*"])` is fully open. `*` is intentionally not a regex ‚Äî operators read these declarations and "any" is the only level of indirection they need. -- alternatives: regex/glob on key names (rejected: hard to read and easy to over-match); enum of pipeline lifecycle phases (rejected: doesn't match the dynamic command-name reality where same handler runs in multiple presets).
- [Implementation] **`PipelineExecutor` resolves the data-flow lazily ‚Äî `IPhaseDataFlowResolver.Resolve(presetName)` returns null when no declaration registered, and the gate is simply not attached.** Tests can construct PipelineExecutor with an empty resolver (`new PhaseDataFlowResolver(Array.Empty<IPhaseDataFlow>())`) and the gate stays dormant ‚Äî every existing test passes unchanged. -- alternatives: throw on missing declaration (rejected: forces every test fixture to register all 10 flows); attach a "deny-all" gate (rejected: same problem).
- [Implementation] **Gate scope wraps SafeExecuteAsync only ‚Äî TrackCommand and post-step bookkeeping read freely.** `context.TrackCommand` reads `ExecutionTrail` and `ActiveSkill`; those are framework-internal keys, not handler-declared inputs. Wrapping only the handler's ExecuteAsync ensures the gate sees handler reads, not infrastructure reads. -- alternatives: gate the entire step including TrackCommand (rejected: would require declaring ExecutionTrail/ActiveSkill on every preset's edges); gate only the inner handler call (rejected: same as chosen approach but spelled differently).
- [TradeOff] **`enforce: false` is the production-config default.** Switching to `true` requires the operator to verify the warning log is empty for their pipeline mix. Until preset edges tighten in follow-up phases, the warning-only mode is the only viable default ‚Äî a hard `enforce=true` on permissive `("*","*","*")` flows would be silent (everything matches the wildcard). The two states (warn-only, strict) both work right now; the value of strict only kicks in once edges tighten. -- alternatives: default to true (rejected: zero benefit until edges are real); ship without the config knob (rejected: operators need a way to opt-in once they've cleaned up the warnings).
- [TradeOff] **No GatedPipelineContext class.** Spec described a wrapper class. Implementation made it a hook on PipelineContext itself. Tests now exercise PipelineContext.AttachReadGate (in `PipelineContextReadGateTests`) plus DataFlowReadGate (in `DataFlowReadGateTests`); the same coverage the spec's wrapper would have had, but spread across two types whose responsibilities are clearly separate (PipelineContext: state bag + gate hook; DataFlowReadGate: enforcement policy). -- alternatives: implement spec verbatim (rejected: would require unsealing PipelineContext or a parallel interface that handlers also have to take).
